# Skill: Environment Management — Runtime Selection + Persistence

**Trigger:** detected by the framework rule §Environment Selection. Loaded automatically when (a) the agent is about to run a stack-runtime command (`pip`, `npm`, `pytest`, `jest`, `go test`, `cargo build`, etc.) AND (b) `.portable-spec-kit/env-config.yml` does not yet have an entry for that stack.

**Purpose:** generic, stack-agnostic interactive flow that detects the project's runtime stack(s), asks the user which environment to use (existing or new), and persists the choice in `.portable-spec-kit/env-config.yml` so every future session + every contributor + every AI agent uses the same env.

---

## Supported stacks + managers

| Stack | Detected by | Supported managers | Default recommendation |
|---|---|---|---|
| **Python** | `requirements.txt` / `pyproject.toml` / `setup.py` / `Pipfile` | conda · venv · poetry · uv · system | conda (kit's historical default per Section §Python Environment) |
| **Node** | `package.json` | nvm · volta · fnm · asdf · system | nvm if installed; system otherwise |
| **Ruby** | `Gemfile` | rbenv · rvm · system | rbenv if installed; system otherwise |
| **Go** | `go.mod` | system · asdf | system (Go handles versioning natively via `go.work` / `toolchain`) |
| **Rust** | `Cargo.toml` | rustup · system | rustup (toolchain config in `rust-toolchain.toml`) |

For other stacks, the kit treats them as `system` (no virtualization) and the user manages versioning manually.

## Workflow (each invocation)

### 1. Detect stacks

```bash
bash agent/scripts/psk-env.sh detect
```

Outputs one stack token per line — `python`, `node`, etc. Empty output means no recognized stack in this project (e.g., a pure-markdown repo) — skip env selection entirely.

### 2. For each detected stack, check if env-config has an entry

```bash
bash agent/scripts/psk-env.sh status
```

If the stack is already in `runtimes:`, the env was already selected in a prior session. Skip to step 5 (verify).

### 3. List available envs (information for the user's choice)

```bash
bash agent/scripts/psk-env.sh list-envs <stack>
```

Outputs lines like `conda:my-env`, `venv:/path/to/.venv`, `system:python3 3.11`. The agent uses this list to render options.

### 4. Ask the user — interactive prompt

Show the user a numbered menu:

```
Project type detected: Python (requirements.txt found)
Where should I run Python commands?

EXISTING envs:
  1. conda: searchsocialtruth (Python 3.11)
  2. conda: base (Python 3.13)
  3. venv: ./.venv (Python 3.10)
  4. system Python (3.13)

CREATE NEW:
  5. New conda env (will name: searchsocialtruth-py311)
  6. New venv (will create: ./.venv with system Python)
  7. New poetry env (uses pyproject.toml)
  8. New uv env (uses pyproject.toml)

Pick [1-8] (default: 1 — recommended for this project):
```

**Default-selection rule:** if a conda env matching the project directory name exists, suggest it as the default. Otherwise default to the most-recently-used manager (read from any other stack's entry in env-config.yml). For a fresh project: default to conda for Python, nvm for Node, rbenv for Ruby.

**No-conda / no-nvm fallback:** if the recommended manager isn't installed, the agent offers to install it (with explicit user confirmation) OR falls back to system. Never silently picks system without telling the user.

### 5. Save the choice

```bash
bash agent/scripts/psk-env.sh set <stack> <manager> <env-name-or-version>
```

Writes the entry to `.portable-spec-kit/env-config.yml`. Idempotent — re-running with different values updates only that stack's entry.

### 6. Verify the env works

```bash
bash agent/scripts/psk-env.sh check
```

Runs a basic version probe through the activation prefix. If broken, the agent surfaces the issue and re-prompts the user (env got deleted, version got removed, etc.).

### 7. Use the env in subsequent commands

For every stack-runtime command, agent prefixes with the activation:

```bash
PREFIX=$(bash agent/scripts/psk-env.sh activate-cmd python)
bash -c "$PREFIX pytest tests/"
bash -c "$PREFIX pip install -r requirements.txt"
```

The script returns an empty string for stacks managed via `system` / `volta` / `asdf` / `rustup` (which handle PATH routing transparently); the prefix-aware bash `-c` invocation works correctly even for empty prefix.

## env-config.yml schema

```yaml
# Generated by agent/scripts/psk-env.sh — do not edit by hand.
schema_version: 1
last_updated: 2026-04-29T12:00:00Z
runtimes:
  python:
    manager: conda           # conda | venv | poetry | uv | system
    env_name: my-project     # for conda
    # OR env_path: /path/to/.venv     (for venv)
    # OR (no extra field for poetry / uv / system)
  node:
    manager: nvm             # nvm | volta | fnm | asdf | system
    version: "20"            # for nvm / fnm
  ruby:
    manager: rbenv
    version: "3.2.0"
  go:
    manager: system
  rust:
    manager: rustup
```

Committed to repo. Every contributor pulls the same env choice; no per-machine `.gitignore` here.

## Edge cases

| Scenario | Handling |
|---|---|
| Multiple stacks in one project (e.g., Next.js + Python ML backend) | Run the workflow once per stack; env-config.yml gets multiple `runtimes:` entries |
| User picks a manager not installed (e.g., `poetry` but no `poetry` binary) | Offer to install it, OR ask user to install + come back, OR pick a different manager |
| Saved env disappears between sessions (e.g., conda env deleted) | `psk-env.sh check` fails → agent surfaces the issue + asks user whether to recreate or pick a different env |
| User wants to switch envs mid-project | Manual: re-run the prompt via `"reset env"` / `"choose env"` triggers, or directly `bash agent/scripts/psk-env.sh set <stack> <new-manager> <new-name>` |
| Project has no recognized stack (pure docs / markdown / bash) | `detect` returns empty; skill is not invoked; no env-config.yml written |
| Cross-platform (macOS vs Linux vs Windows) | Manager names are platform-agnostic; the kit's `build_activate_cmd()` handles platform-specific activation differences (e.g., `source` vs `Activate.ps1`) |

## Backward compatibility with `python-environment.md`

The earlier `python-environment.md` skill (Python-only, Conda-mandatory) is now superseded by this generic skill. For Python projects, this skill recommends conda by default — preserving the prior contract. Older projects with conda envs already running continue to work unchanged; the env-config.yml just makes that choice explicit + portable.

## Performance

- `detect` — 4 file-existence checks, <10ms
- `list-envs` — depends on manager (`conda env list` ~200ms, others <50ms), invoked only at selection time
- `set` — single YAML write, <10ms
- `activate-cmd` — single YAML parse, <10ms

Per-command overhead (kit prefixes a stack command with `activate-cmd`): **<10ms**, negligible compared to the actual command runtime.
